# Plan: True Progressive Streaming for UI & API

## Problem
The current streaming endpoint (`/v1/audio/speech/stream`) accumulates all chunks, cross-fades them, and sends the complete audio at the end. The UI waits for the entire response before playing anything. For long text, the user sees "Generating..." for 10-30+ seconds with no audio feedback.

## Goal
- UI: Audio starts playing the first chunk while remaining chunks are still generating
- API: The `/v1/audio/speech/stream` endpoint sends audio progressively as each chunk completes
- Other API clients (Open WebUI, etc.) can also consume the progressive stream

## Design

### Server-Side: True Progressive PCM Streaming

The `/v1/audio/speech/stream` endpoint currently uses `StreamingResponse` but yields nothing until all chunks are done. Fix it to:

1. **Encode prompt once** (same as now)
2. **Chunk text** (same as now)
3. **For each chunk**: generate audio, encode as raw PCM (int16, 48kHz, mono), **immediately yield** the PCM bytes
4. **No cross-fade** in streaming mode — each chunk's PCM is sent independently as it completes

Content-type: `audio/pcm;rate=48000;format=s16le;channels=1`

This is the same raw PCM format that the non-streaming `response_format=pcm` already produces. Clients that want WAV/MP3 can still use the non-streaming endpoint. Streaming gives raw PCM for lowest latency.

### Why no cross-fade in streaming?
Cross-fade requires having both adjacent chunks available simultaneously to blend the overlap region. In streaming, chunk N is sent before chunk N+1 exists. The 50ms cross-fade is a minor quality improvement at sentence boundaries — the sentence pauses already provide natural transitions. Non-streaming mode retains cross-fade for best quality.

### Client-Side: Web Audio API Progressive Playback

Replace the `response.blob()` + `<audio>` element approach with:

1. **`fetch()` with `response.body.getReader()`** — read the stream progressively
2. **`AudioContext` + `AudioBufferSourceNode`** — decode and play each PCM chunk as it arrives
3. **Chunk scheduling** — each chunk is scheduled to play immediately after the previous one ends (using `AudioContext.currentTime` + the end time of the previous buffer)
4. **Visual progress indicator** — show "Playing chunk 1/5..." and update as each chunk streams

```python
async def audio_generator():
    encoded_prompt = _encode_prompt(voice_file, request.rms, request.ref_duration)
    chunks = _chunk_text(request.input, request.max_chunk_chars)
    
    for i, chunk_text in enumerate(chunks):
        chunk_audio = _generate_single(chunk_text, encoded_prompt, ...)
        # Encode as int16 PCM
        pcm_data = (chunk_audio * 32767).astype(np.int16).tobytes()
        yield pcm_data
```

3. **Web Audio API playback logic**:
```javascript
async function generateSpeechStream() {
    const audioCtx = new AudioContext({ sampleRate: 48000 });
    let nextStartTime = 0;
    
    const response = await fetch('/v1/audio/speech/stream', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ ...requestBody, response_format: 'pcm' })
    });
    
    const reader = response.body.getReader();
    let buffer = new Uint8Array(0);
    const CHUNK_SAMPLES = 48000 * 2; // 2 seconds of int16 audio per buffer
    
    while (true) {
        const { done, value } = await reader.read();
        if (done) break;
        
        // Accumulate into buffer
        buffer = appendBuffer(buffer, value);
        
        // When we have enough samples, decode and schedule
        while (buffer.byteLength >= CHUNK_SAMPLES * 2) {
            const pcmData = buffer.slice(0, CHUNK_SAMPLES * 2);
            buffer = buffer.slice(CHUNK_SAMPLES * 2);
            const audioBuffer = pcmToAudioBuffer(audioCtx, pcmData);
            const source = audioCtx.createBufferSource();
            source.buffer = audioBuffer;
            source.connect(audioCtx.destination);
            
            const now = audioCtx.currentTime;
            const startTime = Math.max(now, nextStartTime);
            source.start(startTime);
            nextStartTime = startTime + audioBuffer.duration;
        }
    }
    
    // Play remaining buffer
    if (buffer.byteLength > 0) {
        const audioBuffer = pcmToAudioBuffer(audioCtx, buffer);
        const source = audioCtx.createBufferSource();
        source.buffer = audioBuffer;
        source.connect(audioCtx.destination);
        source.start(Math.max(audioCtx.currentTime, nextStartTime));
    }
}
```

4. **PCM → AudioBuffer decoder**:
```javascript
function pcmToAudioBuffer(audioCtx, pcmBytes) {
    const int16 = new Int16Array(pcmBytes.buffer, pcmBytes.byteOffset, pcmBytes.byteLength / 2);
    const float32 = new Float32Array(int16.length);
    for (let i = 0; i < int16.length; i++) {
        float32[i] = int16[i] / 32768;
    }
    const buffer = audioCtx.createBuffer(1, float32.length, 48000);
    buffer.copyToChannel(float32, 0);
    return buffer;
}
```

### Streaming Format Decision

**For the UI**: Raw PCM (int16, 48kHz, mono) is the simplest and lowest latency. No encoding overhead per chunk. The Web Audio API can directly consume float32 samples derived from int16.

**For other API clients**: Also raw PCM. This is the most compatible progressive streaming format. Clients who want WAV/MP3 should use the non-streaming endpoint (`stream=false`). We document this clearly.

### Edge Cases

1. **Single chunk (short text)**: Streaming still works — single chunk is sent and plays immediately
2. **Chunking disabled (`max_chunk_chars=0`)**: Entire text is one chunk, still streams
3. **CUDA OOM mid-stream**: The server can send an error in the stream or close the connection. The UI should handle connection errors gracefully.
4. **AudioContext autoplay**: Browsers require user gesture to create AudioContext. The "Generate" button click is the gesture, so this is fine.
5. **Memory on client**: Int16 PCM at 48kHz = 96KB/second. A 30-second clip = ~2.9MB — trivial for browser memory.

### What about cross-fade quality?

In streaming mode, there's no cross-fade. The sentence boundary pauses in TTS output make this mostly seamless. For users who want perfect quality, the non-streaming endpoint retains cross-fade. This is an acceptable tradeoff — streaming prioritizes latency, non-streaming prioritizes quality.
